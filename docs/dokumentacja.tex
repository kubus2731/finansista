\section{Wstęp}
\label{sec:wstep}

\subsection{Cel projektu}
% TODO: 3-5 zdań - po co system, dla kogo, jaki realny problem rozwiązuje.
Celem projektu jest dostarczenie systemu informatycznego usprawniającego proces
składania, oceny i rozliczania wniosków o finansowanie działalności studenckiej
i doktoranckiej w Politechnice Białostockiej. System realizuje papierowy obieg
wniosku opisany w \emph{Zarządzeniu nr 13/2025 Rektora Politechniki Białostockiej
z dnia 14 lutego 2025 r. w sprawie wydatkowania środków finansowych przeznaczonych
na działalność studencką i doktorancką}, ze szczególnym uwzględnieniem
\emph{Załącznika nr 1} (wzór formularza wniosku).

\subsection{Zakres}
System obejmuje:
\begin{itemize}[leftmargin=*]
  \item rejestrację oraz logowanie użytkowników z rozróżnieniem ról,
  \item składanie wniosku w wersji elektronicznej w pełni odwzorowującej Załącznik nr 1,
  \item walidację formularza (pola obowiązkowe, spójność dat i kwot, suma źródeł finansowania),
  \item obieg statusów wniosku (maszyna stanów po stronie bazy danych),
  \item podgląd statusu i szczegółów wniosku przez wnioskodawcę,
  \item REST API jako warstwę komunikacji frontend--backend,
  \item dokumentację OpenAPI dla wystawionych usług,
  \item logikę składowaną w bazie danych (pakiety, widoki, wyzwalacze, funkcje),
  \item zgodność z wymaganiami dostępności (deklaracja dostępności).
\end{itemize}

\subsection{Zespół i podział pracy}
\begin{itemize}[leftmargin=*]
  \item \textbf{Jakub Matusiewicz} -- warstwa bazy danych (Oracle, PL/SQL, migracje Flyway),
    frontend (Thymeleaf, Bootstrap), integracja widoków z REST API.
  \item \textbf{Jakub Borkowski} -- backend Spring Boot, model encji JPA, use case'y,
    REST API, bezpieczeństwo (Spring Security + JWT).
\end{itemize}

\section{Wymagania}
\label{sec:wymagania}

\subsection{Wymagania funkcjonalne}
% Lista z deklaracji - możesz dopisać/skreślić wg stanu.
\begin{enumerate}[leftmargin=*]
  \item Rejestracja i logowanie użytkownika.
  \item Tworzenie wniosków w wersji elektronicznej.
  \item Wypełnianie wniosku poprzez formularz odwzorowujący Załącznik nr 1.
  \item Walidacja formularza.
  \item Wysyłanie wniosku do wyższych jednostek (zmiana statusu DRAFT $\to$ SUBMITTED).
  \item Kontrola wniosku finansowego poprzez sprawdzanie jego statusu.
  \item Edycja/poprawa wniosku w razie odrzucenia lub uwag.
  \item Procedowanie wniosku przez wyższe jednostki (Dziekanat, CSSDiR, Kwestor) -- pola
        dostępne dla odpowiednich ról oraz odpowiednie ścieżki dla konkretnych wniosków.
  \item Dostępność aplikacji dla osób ze specjalnymi potrzebami.
  \item Filtrowanie i wyszukiwanie wniosków.
  \item Zarządzanie uprawnieniami i rolami użytkowników (administrator).
  \item Wyświetlanie pełnej historii zmian statusu wniosku.
  \item Źródło finansowania jako zmienna w encji wniosku.
\end{enumerate}

\subsection{Wymagania niefunkcjonalne}
\begin{itemize}[leftmargin=*]
  \item \textbf{Architektura}: klient--serwer; frontend działa niezależnie od backendu
        i komunikuje się z nim wyłącznie poprzez REST API.
  \item \textbf{Wydajność}: poprawna praca przy zbiorach 1\,000--1\,000\,000 wniosków
        (z paginacją i odpowiednimi indeksami).
  \item \textbf{Bezpieczeństwo}: uwierzytelnianie tokenem JWT przechowywanym w bezpiecznym
        ciasteczku \texttt{HttpOnly}, autoryzacja oparta na rolach.
  \item \textbf{Dostępność}: zgodność z WCAG 2.1 -- skip link, semantyczne nagłówki,
        kontrast, etykiety formularzy, opublikowana deklaracja dostępności.
\end{itemize}

\section{Architektura systemu}
\label{sec:architektura}

System jest aplikacją monolityczną logicznie podzieloną na trzy warstwy:

\begin{itemize}[leftmargin=*]
  \item \textbf{Warstwa prezentacji (frontend)} -- kontrolery Spring MVC zwracające
        widoki Thymeleaf renderowane po stronie serwera. Frontend \emph{nie woła}
        warstwy biznesowej bezpośrednio -- komunikuje się z backendem wyłącznie
        przez REST API z użyciem \texttt{RestClient}, przekazując token JWT.
  \item \textbf{Warstwa REST API (backend)} -- kontrolery \texttt{@RestController}
        wystawiające usługi pod ścieżką \texttt{/api/v1/...} (JSON). Stanowi
        kontrakt komunikacyjny niezależny od użytego klienta.
  \item \textbf{Warstwa danych} -- baza Oracle z migracjami Flyway, logiką
        składowaną (PL/SQL) oraz dostępem przez JPA/Hibernate.
\end{itemize}

\subsection{Główne tabele}
\begin{itemize}[leftmargin=*]
  \item \texttt{users} -- konta użytkowników; powiązane z \texttt{roles} i \texttt{department}.
  \item \texttt{requests} -- główna encja wniosku (sekcje I--III Załącznika nr 1).
  \item \texttt{request\_task} -- pozycje harmonogramu (sekcja IV).
  \item \texttt{request\_cost\_item} -- pozycje kosztorysu (sekcja IV).
  \item \texttt{request\_funding} -- źródła finansowania z kwotą wnioskowaną i przyznaną (sekcja VI).
  \item \texttt{request\_status} -- słownik statusów wniosku.
  \item \texttt{activity\_log} -- historia zmian statusu.
  \item \texttt{comments}, \texttt{attachments} -- komentarze i załączniki do wniosków.
  \item Słowniki: \texttt{roles}, \texttt{department}, \texttt{cost\_category},
        \texttt{funding\_source}, \texttt{request\_templates}.
\end{itemize}

\subsection{Klucze hybrydowe}
Każda eksponowana encja używa dwóch identyfikatorów:
wewnętrznego \texttt{id} (\texttt{NUMBER IDENTITY}) używanego w kluczach obcych
oraz zewnętrznego \texttt{external\_id} (\texttt{RAW(16)} -- UUID v7),
który jest jedynym identyfikatorem wystawianym w REST API i adresach URL.
Zapobiega to wycieksowi informacji o liczności rekordów i utrudnia enumerację zasobów.

\subsection{Migracje Flyway}
\begin{itemize}[leftmargin=*]
  \item \texttt{V1} -- inicjalny schemat (11 tabel: users, roles, department, requests, \ldots).
  \item \texttt{V2} -- wypełnienie słowników (role, działy, statusy, kategorie).
  \item \texttt{V3} -- widoki bazodanowe (m.in. \texttt{v\_department\_requests\_summary}).
  \item \texttt{V4} -- wyzwalacze (audyt, walidacje).
  \item \texttt{V5} -- pakiet PL/SQL \texttt{pkg\_finansista} (procedury domenowe).
  \item \texttt{V6} -- indeksy wydajnościowe.
  \item \texttt{V7} -- maszyna stanów wniosku (funkcja + trigger blokujący niedozwolone przejścia).
  \item \texttt{V8} -- rozszerzenie schematu pod Załącznik nr 1 (sekcje I--III + tabele
        \texttt{request\_task}, \texttt{request\_cost\_item}, \texttt{request\_funding}).
  \item \texttt{V9} -- dodanie jednostki \emph{Samorząd Doktorantów PB}.
  \item \texttt{V10} -- kategoria \emph{Inne} + pole opisowe \texttt{cost\_category\_other}.
\end{itemize}

\section{REST API}
\label{sec:api}

Frontend Thymeleaf komunikuje się z backendem wyłącznie poprzez REST API, co
wymusza poprawną architekturę klient--serwer. Token JWT z ciasteczka przeglądarki
jest przekazywany jako nagłówek \texttt{Authorization: Bearer}.

\subsection{Najważniejsze endpointy}
\begin{tabularx}{\textwidth}{l l X}
  \toprule
  \textbf{Metoda} & \textbf{Ścieżka} & \textbf{Opis} \\
  \midrule
  POST & /api/v1/auth/register & rejestracja konta \\
  POST & /api/v1/auth/login & logowanie (ustawia ciasteczko JWT) \\
  POST & /api/v1/auth/logout & wylogowanie (kasuje ciasteczko) \\
  GET  & /api/v1/requests & lista wniosków (filtr po statusie/wydziale) \\
  POST & /api/v1/requests & utworzenie wniosku (pełny Załącznik nr 1) \\
  GET  & /api/v1/requests/\{id\} & szczegóły wniosku \\
  PUT  & /api/v1/requests/\{id\} & edycja wniosku \\
  PATCH & /api/v1/requests/\{id\}/status & zmiana statusu \\
  GET  & /api/v1/requests/\{id\}/history & historia zmian statusu \\
  POST & /api/v1/requests/\{id\}/comments & dodanie komentarza \\
  GET  & /api/v1/requests/\{id\}/comments & komentarze \\
  DELETE & /api/v1/requests/\{id\} & usunięcie wniosku (admin) \\
  \bottomrule
\end{tabularx}

\subsection{Specyfikacja OpenAPI}
Pełna specyfikacja jest udostępniona przez Swagger UI pod adresem
\url{http://localhost:8080/swagger-ui/index.html} oraz w formie JSON pod
\url{http://localhost:8080/v3/api-docs}.

\section{Bezpieczeństwo}
\label{sec:bezp}

\subsection{Uwierzytelnianie}
Po poprawnym zalogowaniu backend wystawia token JWT podpisany kluczem HMAC SHA-256
i zwraca go w bezpiecznym ciasteczku z atrybutami \texttt{HttpOnly}, \texttt{SameSite=Strict}.
Token zawiera identyfikator użytkownika (e-mail) oraz jego rolę.

\subsection{Autoryzacja oparta na rolach}
\begin{itemize}[leftmargin=*]
  \item \texttt{ROLE\_STUDENT} -- składa wnioski, widzi własne;
  \item \texttt{ROLE\_WRSS} -- ocena merytoryczna w ramach Samorządu;
  \item \texttt{ROLE\_LEGAL\_COMMISSION} -- ocena merytoryczna w komisji;
  \item \texttt{ROLE\_DEAN\_OFFICE} -- ocena formalna (CSSDiR/Dziekanat);
  \item \texttt{ROLE\_FINANCE\_OFFICE} -- Kwestura (rozliczenie);
  \item \texttt{ROLE\_ADMIN} -- decyzja Rektora, zarządzanie systemem.
\end{itemize}

\subsection{Zabezpieczenia formularzy}
Rola użytkownika \emph{nigdy} nie jest pobierana z formularza rejestracji --
każde nowe konto otrzymuje rolę bazową \texttt{ROLE\_STUDENT} wymuszoną po stronie
serwera. Przypisanie wyższej roli jest wyłącznie aktem administracyjnym.
Podobnie wydział wnioskodawcy nie jest wybierany przy składaniu wniosku --
pobierany jest z konta zalogowanego użytkownika.

\end{itemize}
